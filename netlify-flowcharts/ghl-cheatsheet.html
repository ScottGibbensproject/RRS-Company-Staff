GoHighLevel (GHL) Documentation

Operational guides for day-to-day tasks, training courses, and system configuration

📝Common Tasks Cheat Sheet

Written for day-to-day operators (you, Elizabeth, resellers later). Avoids theory and focuses on where to click and what actually works.

1. Create a Contact Form

Use case: Capture leads, reseller signups, demo requests.

  1. Go to Sites → Forms
  2. Click Create Form
  3. Choose Blank Form
  4. Add fields (Name, Email, Phone, Custom Fields as needed)
  5. Click Styles & Options to adjust layout
  6. Save the form

Best Practices

  • Keep forms short (3-5 fields max)
  • Phone number first for SMS workflows
  • Use custom fields only when necessary

2. Assign Tags to a Contact Form (Critical)

Use case: Automatically segment contacts when they submit a form.

  1. Open the form
  2. Click Options (right panel)
  3. Scroll to On Submit
  4. Add Tag(s) (example: reseller-lead, demo-request)
  5. Save

Why this matters

  • Tags trigger workflows
  • Tags drive Smart Lists
  • Tags are more reliable than form names

3. Create a Workflow

Use case: Onboarding, follow-ups, internal tasks, automation.

  1. Go to Automation → Workflows
  2. Click Create Workflow
  3. Choose Start from Scratch
  4. Add a Trigger (e.g., Tag Added, Form Submitted)
  5. Add Actions (Send Email, Add Tag, Create Task, Assign User)
  6. Set delays if needed
  7. Publish workflow

Workflow Rules

  • One workflow = one purpose
  • Name workflows clearly
  • Never stack unrelated logic in one workflow

4. Create an Email Campaign / Template (Correct Way)

Use case: One-time announcements, newsletters, branded bulk email.

  1. Go to Marketing → Emails
  2. Click Create Email
  3. Choose Blank Email or a saved template
  4. Write content or paste approved HTML
  5. Save as Template if reusable

Important

  • Do NOT use Contacts → Bulk Email for branded sends
  • Marketing Emails support HTML, templates, signatures

5. Send a One-Time Bulk Email (Proper Method)

  1. Go to Marketing → Emails
  2. Create or open an email
  3. Select Send to Contacts / Smart List
  4. Choose recipients
  5. Send immediately or schedule

Tip: Smart Lists > manual selection every time.

6. Assign a Contact to a User (Owner)

Use case: Sales ownership, reseller visibility, accountability.

Manual Assignment

  1. Open a contact record
  2. Click Assigned To
  3. Select user
  4. Save

Automation Assignment

  1. In a workflow, add Assign User action
  2. Choose specific user or round-robin

7. Create Smart Lists

Use case: Segment contacts dynamically.

  1. Go to Contacts
  2. Apply filters (Tag, Field, Owner, Status)
  3. Click Save as Smart List
  4. Name clearly

Rules

  • Smart Lists are filter-based, not manual
  • They auto-update

8. Create Custom Fields

Use case: Store structured data (Reseller Stage, Territory, Status).

  1. Go to Settings → Custom Fields
  2. Click Add Field
  3. Choose field type (Single Line, Dropdown, Checkbox)
  4. Name clearly
  5. Save

Best Practices

  • Use Dropdowns for stages/statuses
  • Avoid free-text where consistency matters

9. Add Tasks Inside Workflows

Use case: Internal reminders, onboarding steps that can't be automated.

  1. In workflow, add Create Task action
  2. Name task clearly
  3. Assign to a user
  4. Set due date or delay

Example: "Create demo merchant account"

10. Conversations & Internal Notes

Use case: Customer communication vs internal discussion.

Rules

  • Conversations = customer-facing
  • Internal Notes = team-only
  • Visibility depends on assignment

Important

Users set to "View own activity only" only see assigned conversations.

11. When to Use What (Quick Reference)

Forms → Capture data
Tags → Trigger logic
Workflows → Automation
Marketing Emails → Bulk branded email
Bulk Email (Contacts) → Plain-text only
Smart Lists → Dynamic segmentation
Custom Fields → Structured data

Final Rule of Thumb:

If it needs to scale or repeat → use a workflow.
If it needs branding → use Marketing Emails.
If it needs tracking → use tags.

🎓Training Courses Setup Guide

How to structure, build, and manage multiple training courses in GoHighLevel for different audiences.

1. Core Concept (Read This First)

In GHL:

  • Courses = containers
  • Lessons = content pages (video, text, links)
  • Quizzes = gatekeepers
  • Tags = permissions + progress tracking
  • Workflows = enforcement + automation
Access is controlled by tags, not by users or roles.

2. Recommended Course Structure

Course 1: Direct Agent Training

Audience: Your in-house / direct agents

Purpose:

  • Sales process
  • Software overview
  • Onboarding merchants
  • Internal standards

Access Tag: course-direct-agent

Course 2: ISO Group Training (Standard)

Audience: ISO partners selling under your program

Purpose:

  • Product overview
  • Compliance
  • Sales positioning
  • Merchant onboarding (limited)

Access Tag: course-iso-standard

Course 3: ISO Group Training (Advanced)

Audience: Approved / experienced ISO partners

Purpose:

  • Advanced sales scenarios
  • Territory strategy
  • Deeper software features
  • Revenue optimization

Access Tag: course-iso-advanced

Prerequisite Tag: iso-certified

3. Create a Course

  1. Go to Sites → Memberships → Courses
  2. Click Create Course
  3. Name the course clearly (include audience)
  4. Add description (internal clarity matters)
  5. Save

Naming Convention (Important)

  • Direct Agent - Core Training
  • ISO Training - Standard
  • ISO Training - Advanced

4. Create Lessons

  1. Open a course
  2. Click Add Lesson
  3. Add:
    • Video (preferred)
    • Supporting text
    • Links / downloads
  4. Save

Best Practices

  • One concept per lesson
  • Keep lessons under 15 minutes
  • Avoid dumping PDFs without context

5. Create Quizzes (Critical for Control)

Purpose: Enforce completion, prevent skipping, trigger certification.

  1. Go to Sites → Memberships → Quizzes
  2. Create quiz
  3. Add questions (multiple choice recommended)
  4. Set passing score (ex: 80%)
  5. Save

6. Lesson → Quiz Gating (Required)

Correct Flow

Lesson → Quiz → Next Lesson

Implementation

  • Place quiz immediately after key lessons
  • Do NOT rely on "mark complete" alone

7. Tag-Based Access Control

Grant Course Access (Workflow Example)

Trigger: Tag added course-direct-agent

Action: Grant access to Direct Agent - Core Training

Repeat for each course.

Remove / Upgrade Access (Example)

When iso-certified tag is added:

  • Grant access to ISO Training - Advanced

8. Certification Logic (Very Important)

How Certification Works

  1. Final quiz passed
  2. Workflow fires
  3. Tag applied: certified
  4. Unlock next permissions
Never rely on manual certification.

9. Workflow Examples You Should Have

A. Course Enrollment Workflow

Trigger: Tag added

Actions:

  • Grant course access
  • Send welcome email

B. Quiz Passed Workflow

Trigger: Quiz passed

Actions:

  • Add tag certified
  • Send confirmation email
  • Notify internal team

C. Advanced ISO Unlock Workflow

Trigger: Tag iso-certified added

Actions:

  • Grant advanced course access
  • Notify account owner

10. What NOT to Do

Avoid These Mistakes

  • Do not create one giant course
  • Do not manually enroll users long-term
  • Do not mix Direct and ISO content
  • Do not skip quizzes

11. Reporting & Visibility

Use:

  • Tags → completion status
  • Smart Lists → who is certified / stuck
  • Tasks → follow-up on incomplete training

12. Recommended Tags (Standardize These)

course-direct-agent course-iso-standard course-iso-advanced certified iso-certified training-incomplete

Final Rule:

Courses teach. Quizzes enforce. Tags decide access. Workflows run the business.

📝Lesson Naming & Numbering Standard

Prevents confusion as courses grow. Follow this exactly. Do not improvise.

1. Why This Matters

Without a strict naming system:

  • Support tickets become unclear ("Lesson 3? Which one?")
  • Workflows break when lessons are reordered
  • New staff cannot help users efficiently
  • Advanced ISO content bleeds into standard training

Consistency here saves real time later.

2. Core Naming Formula

Lesson Name Format:

[Course Code] - [Section #].[Lesson #] - [Short Title]

3. Course Codes (Standardized)

Use these exact prefixes:

  • DA → Direct Agent
  • ISO-S → ISO Standard
  • ISO-A → ISO Advanced

Examples:

  • DA - Direct Agent - Core Training
  • ISO-S - ISO Training - Standard
  • ISO-A - ISO Training - Advanced

4. Lesson Numbering Rules

  • Sections increment by whole numbers (1, 2, 3...)
  • Lessons increment within sections (1.1, 1.2, 1.3...)
  • Never renumber once published
  • Add new lessons using decimals (1.4, 1.5)

This avoids breaking references and workflows.

5. Examples (Correct)

Direct Agent Course

  • DA - 1.1 - Introduction & Expectations
  • DA - 1.2 - How the Program Works
  • DA - 1.3 - Compliance Overview
  • DA - 2.1 - Software Walkthrough
  • DA - 2.2 - Creating a Merchant Demo

ISO Standard Course

  • ISO-S - 1.1 - Program Overview
  • ISO-S - 1.2 - Sales Positioning
  • ISO-S - 1.3 - Merchant Onboarding Rules
  • ISO-S - 2.1 - Software Basics

ISO Advanced Course

  • ISO-A - 1.1 - Advanced Sales Scenarios
  • ISO-A - 1.2 - Territory Strategy
  • ISO-A - 2.1 - Advanced Software Features

6. Quiz Naming Standard

Quiz Name Format:

[Course Code] - Quiz - [Section #]

Examples:

  • DA - Quiz - Section 1
  • ISO-S - Quiz - Section 2
  • ISO-A - Final Certification Quiz

7. Workflow & Tag Alignment (Critical)

  • Workflows should reference quiz names, not lesson names
  • Certification workflows should only trigger on Final Certification Quiz

Recommended tags:

quiz-passed-section-1 quiz-failed-section-2 certified iso-certified

8. What NOT to Do

Avoid These Mistakes

  • Do not rename lessons after users enroll
  • Do not skip numbering
  • Do not reuse lesson titles across courses
  • Do not rely on lesson order alone

Operational Rule:

If someone asks a question, they must be able to quote the lesson name exactly. If that's not possible, the naming system is wrong.

🔄Course Versioning Strategy

How to update courses over time without breaking progress, certifications, or trust.

1. Core Rule (Non-Negotiable)

Never materially change a live course that people have already completed.

Instead, version it.

2. What Triggers a New Version

Create a new version when you:

  • Change compliance rules
  • Change sales positioning
  • Add or remove required steps
  • Change certification criteria
  • Add material that would change how someone operates

Minor edits (typos, clarity, video trims) do not require versioning.

3. Version Naming Standard

Append the version to the course name:

  • Direct Agent - Core Training (v1)
  • ISO Training - Standard (v2)
  • ISO Training - Advanced (v1)

Never overwrite a version number.

4. Tag-Based Version Control

Each version gets its own access tag.

course-direct-agent-v1 course-iso-standard-v2 course-iso-advanced-v1

Rules

  • Old versions stay accessible to those enrolled
  • New enrollments get the latest version only

5. Certification Tag Rules Across Versions

Certification tags do not change across versions unless the program rules change.

Example: certified remains valid for v1 and v2

If certification rules materially change:

  • Introduce a new tag: certified-v2
  • Do NOT remove old certification tags

6. Workflow Design for Versioning

Enrollment Workflow (Example)

Trigger: New reseller / agent created

Actions:

  • Add tag: course-iso-standard-v2
  • Grant access to ISO Training - Standard (v2)

Never reference course names directly in logic without version tags.

7. Upgrading Existing Users (Optional, Controlled)

If you want existing users to take the new version:

  1. Create a separate upgrade workflow
  2. Notify the user clearly
  3. Add the new version tag
  4. Do NOT remove old access automatically

This preserves history.

8. Archiving Old Versions

When a version is fully deprecated:

  • Remove it from menus
  • Keep it accessible only via tag
  • Do NOT delete courses

Deletion breaks records and audits.

9. Reporting & Visibility

Recommended Smart Lists:

  • Course v1 - Active Users
  • Course v2 - Active Users
  • Certified - Legacy
  • Certified - Current

This lets you see who is trained on what rules.

10. What NOT to Do

Avoid These Mistakes

  • Do not edit quizzes tied to certification
  • Do not rename live courses without versioning
  • Do not reuse tags across versions
  • Do not force re-training silently

Final Rule:

Training history is part of your compliance record. Treat it like accounting, not marketing.

🔒Permission / Role Matrix

Defines who can see what, do what, and break what inside GoHighLevel.

Rule: Permissions are set once, roles are reused forever. Do not customize per user unless there is a documented exception.

1. Role Definitions (Canonical)

A. Internal Staff (Admin)

Who: You, Elizabeth, core operations

Purpose: System management, support, approvals

B. Direct Agents

Who: In-house sales agents working directly for Rate Remover

Purpose: Prospecting, onboarding merchants, limited CRM usage

C. ISO Partners (Standard)

Who: External partners selling under your program

Purpose: Deal referral, limited visibility, training access

D. ISO Partners (Advanced)

Who: Approved ISOs with advanced privileges

Purpose: Expanded selling authority, advanced training, deeper reporting

2. High-Level Permission Matrix

Feature Area Staff (Admin) Direct Agent ISO Standard ISO Advanced
Contacts Full Assigned only Assigned only Assigned only
Conversations All Assigned only Assigned only Assigned only
Opportunities / Pipelines Full Assigned only View / Assigned View / Assigned
Workflows Full None None None
Campaigns / Marketing Emails Full None None None
Courses / Training Manage Consume Consume Consume
Quizzes Manage Take Take Take
Custom Fields Full None None None
Settings Full None None None
Billing Full None None None
Reports Full Limited Limited Limited

3. Detailed Permission Settings by Role

A. Staff (Admin)

Enable:

  • View all contacts
  • View all conversations
  • Manage workflows
  • Manage campaigns
  • Manage courses and quizzes
  • Manage custom fields
  • Manage settings
  • View all reports

Disable: Nothing (Admins are trusted)

B. Direct Agent

Enable:

  • View assigned contacts only
  • View assigned conversations only
  • Create and update contacts
  • Create opportunities (assigned)
  • Access training courses

Disable:

  • View all contacts
  • Workflows
  • Campaigns
  • Custom fields
  • Settings
  • Billing

Key Setting: User visibility: View own activity only

C. ISO Partner (Standard)

Enable:

  • View assigned contacts only
  • View assigned conversations only
  • View opportunities (assigned)
  • Access ISO Standard training

Disable:

  • Create workflows
  • Edit campaigns
  • Edit contacts globally
  • Settings
  • Billing

Key Setting: No access to internal pipelines unless explicitly assigned

D. ISO Partner (Advanced)

Enable:

  • Same as ISO Standard
  • Access Advanced ISO training
  • Expanded reporting (read-only)

Disable: Same restrictions as ISO Standard

4. Assignment Rules (Critical)

  • Conversations are visible only if assigned
  • Internal notes are visible only to users who can see the conversation
  • Never assign internal threads to agents or ISOs

5. Automation Guardrails

Workflow Rules

  • Never auto-assign internal conversations to non-staff
  • Never grant permissions via workflows
  • Permissions are set manually, access is granted via tags

6. Training Access vs System Access (Important Distinction)

  • Training access is controlled by tags
  • System permissions are controlled by roles

Never mix the two.

7. Onboarding Checklist (Permissions)

When adding any user:

  1. Assign correct role
  2. Set visibility to "View own activity only" (non-staff)
  3. Verify conversations access
  4. Add appropriate training tags
  5. Confirm they cannot see settings

8. What NOT to Do

Avoid These Mistakes

  • Do not give ISOs admin access
  • Do not let agents edit workflows
  • Do not rely on trust instead of permissions
  • Do not grant exceptions without documentation

Final Rule:

If someone can break something, they eventually will. Lock it down first.
